## Decisions (agreed with David on the bus, #206)
- Q1 intro wording: "This is {Show}. Episode {N}. {Title}. Published on the {Nth of Month YYYY}. This episode is about {summary}." Number + date only when the feed provides them. Summary content unchanged.
- Q2 chime: speed + volume-normalize the SPEECH to match the episode; keep the 0.5s chime as a CONSTANT marker (unchanged length/level).
- Q3 boot: boot on the QUEUE (up-next) EVERY time; never restore a saved route on boot.
- Q4 process: 2 lanes + design gate (done) + 1 pre-commit gate, then real-app verify.

## The three issues (root cause)
1. INTRO METADATA. `announce.cjs buildAnnouncementText` uses show + title + LLM summary only. Publish date is already on each item (`pocketcastsAdapter.js:41`, raw `ep.published`) but never threaded in. Episode/season number not captured (available from podcast/full `ex.episode`/`ex.season` for feeds that number; narrative shows like RiC don't).
2. INTRO SPEED+VOLUME. `converter.cjs` deliberately exempts the intro from atempo(speed) + BOOST (introChain = [normalise] only). That is why the intro was slow+quiet vs the fast+loud episode.
3. BOOT FLASH. `App.jsx:31` restores `os_route` from localStorage verbatim, so a cold boot can land on the transient "syncing" transfer screen, paint empty/low, then settle once data loads.

## GPT-5's catch (folded in): CACHE INVALIDATION
`sync.cjs` variant key (~line 655) = `${speedSuffix}${boostSuffix}${introSuffix}${cutSuffix}` where `introSuffix` is a boolean "-intro". Changing the intro TEXT (issue 1) or PROCESSING (issue 2) does not change the key, so a stale cached mp3 (old intro) would be reused. Must bump an intro/pipeline VERSION token (and/or hash the intro content) into the key so pre-fix encodes are invalidated.

## Lanes (sequential, one branch `fix/swimcast-3issues`, disjoint files)

Format for SPEECH (UTC date components, no ISO/numeric/S03E07), omit absent fields. Boot route always "up-next"; tidy the idle SyncScreen empty-state. Bump the cache version token in sync.cjs.
